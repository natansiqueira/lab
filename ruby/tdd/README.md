# Test-Driven Development

Allow the developer to introduce new features reliably. The idea is to create a failing test/behavior before implementing the code that satisfies the test conditions

1. Write a failing test/behavior
1. Make it pass
1. Make it better

We must follow this cycle for each new feature we want to implement

## Pros
* Reliability
* Avoid the fear of changing the implementation
* Refactoring is safer as we have a way to identify problems in our code as soon as we have a failing test

## Cons
* We have to write extra code for tests
* More code to maintain
* Too large tests or trivial tests
